## 2Ô∏è‚É£ What We Did in **Phase‚ÄØ1 ‚Äì Setup & Index Creation**

### 2.1 Code Files Created (see `src/ingestion/`)
| File | Purpose | Key Functions |
|------|---------|----------------|
| `document_loader.py` | Load PDFs from a directory | `load_documents(input_dir: str) -> List[Document]` ‚Äì uses `SimpleDirectoryReader` from **LlamaIndex** |
| `chunker.py` | Split loaded documents into semantic chunks | `chunk_documents(documents: List[Document]) -> List[BaseNode]` ‚Äì uses `SentenceSplitter(chunk_size=512, chunk_overlap=50)` |
| `embedder.py` | Provide an embedding model | `get_embedder() -> HuggingFaceEmbedding` ‚Äì wraps **intfloat/e5‚Äëlarge‚Äëv2** (1024‚Äëdimensional) |
| `build_index.py` | End‚Äëto‚Äëend pipeline that creates a FAISS vector store | `main()` ‚Äì sets `Settings.embed_model`, loads docs, chunks them, builds `FaissVectorStore`, persists to `data/llamaindex_storage/` |

### 2.3 Rationale Behind Technology Choices (IEEE‚Äëstyle justification)
- **Open‚Äësource stack** ‚Äì All components are free, reproducible, and can run on commodity hardware (CPU‚Äëonly). This satisfies the ‚Äúcost‚Äëeffective‚Äù constraint in the SPEC.
- **LlamaIndex** ‚Äì Provides a unified API for document loading, chunking, embedding, and vector‚Äëstore integration, dramatically reducing boiler‚Äëplate and ensuring future extensibility (e.g., swapping out the embedder).
- **E5‚ÄëLarge‚ÄëV2** ‚Äì State‚Äëof‚Äëthe‚Äëart open‚Äësource embedding model with strong performance on retrieval benchmarks (MTEB). It produces 1024‚Äëdimensional vectors, which align perfectly with FAISS‚Äôs default inner‚Äëproduct metric.
- **FAISS** ‚Äì Industry‚Äëstandard similarity search library; works offline, has sub‚Äëmillisecond query latency on a laptop, and is widely cited in academic RAG papers.
- **Virtual environment & pinned `requirements.txt`** ‚Äì Guarantees deterministic builds for reviewers and reproducibility for the conference submission.

## What Happens When You Run **`/execute 2`** (Phase‚ÄØ2 ‚Äì RAG Pipeline)

In plain, non‚Äëtechnical language, here‚Äôs what the system will do step by step:

1. **Turn on the brain** ‚Äì The code loads a small, fast version of the BioMistral language model (a 4‚Äëbit AI that can write answers). It runs on your regular CPU, so no special hardware is needed.

2. **Open the library of knowledge** ‚Äì It reads the FAISS index we built in Phase‚ÄØ1. This index is just a very smart list of all the sentences from your medical PDFs, organized so we can find the right ones instantly.

3. **Match your question to the right text** ‚Äì When you ask a medical question, the system first looks in the FAISS index and pulls the most relevant textbook passages.

4. **Ask the brain to answer** ‚Äì The retrieved passages are handed to BioMistral, which writes a concise answer that is based only on those passages (so it can‚Äôt hallucinate unrelated facts).

5. **Remember the conversation** ‚Äì (Later in this phase) we‚Äôll add a tiny memory component that keeps track of what you asked before, so follow‚Äëup questions like ‚ÄúWhat about the side‚Äëeffects?‚Äù make sense.
